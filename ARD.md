# Architecture Requirements Document

## 🧱 System Overview
SnapForge is a local-first macOS 15+ screen capture studio built entirely in Swift 6 and SwiftUI 6. It combines ScreenCaptureKit-based capture with on-device AI inference via Core ML and Apple Vision framework, a SwiftData-backed capture library with SQLite FTS5 full-text search, and a Metal-accelerated recording pipeline. The app runs as a menu bar utility with NSWindow-customized overlays for capture, annotation, and a command palette launcher. All AI processing runs on-device by default with a pluggable backend architecture supporting local inference engines (MLX, llama.cpp, Ollama) and optional cloud APIs (OpenAI, Anthropic). No server is required for core functionality.

## 🏗 Architecture Style
Modular monolith with service-oriented internal architecture. The app is a single macOS process organized into distinct Swift packages/modules communicating through protocol-based interfaces and Swift Structured Concurrency. State flows unidirectionally from services through the Observation framework into SwiftUI views. The AI subsystem uses a pluggable provider pattern behind a unified inference protocol, with a model manager handling download, cache, and lifecycle. Streaming inference results flow as AsyncSequence from backend through service layer to UI.

## 🎨 Frontend Architecture
- **Framework:** SwiftUI 6 targeting macOS 15+ (Sequoia). NSWindow customization for borderless capture overlays, command palette, and post-capture action bar. Menu bar scene as primary app entry point with Settings scene for configuration. Uses .ultraThinMaterial and .regularMaterial for translucent overlay panels. matchedGeometryEffect for smooth transitions between capture states. PhaseAnimator for post-capture action bar animations and progressive disclosure drawer. TimelineView for real-time recording indicators (CPU/GPU usage, duration, FPS counter).
- **State Management:** Observation framework (@Observable) for all view models and service state. Each major feature area (Capture, Library, AI, Sharing, Automation) has an @Observable service object vended from a central AppServices container. SwiftUI views observe these directly — no Combine publishers or ObservableObject needed. Capture state machine modeled as a Swift enum with associated values, driven by async/await transitions. User preferences stored in @AppStorage with a strongly-typed PreferencesService wrapper.
- **Routing:** Single-window menu bar app with programmatic NSWindow management for overlays. No traditional navigation stack. Command palette presented as a floating NSPanel. Post-capture action bar presented as a HUD-style NSWindow anchored to capture region. Library view presented in a standard NSWindow with sidebar navigation (capture types, tags, smart filters). Settings scene uses native SwiftUI Settings with TabView. Window lifecycle managed by a WindowManager service using Structured Concurrency for coordinating overlay presentation and dismissal.
- **Build Tooling:** Swift Package Manager for modular internal architecture. The app is structured as a root Xcode project with SPM local packages: CaptureEngine, AIEngine, LibraryStore, SharingService, AutomationBridge, CommonUI. Each package declares explicit dependencies. Core ML models bundled as .mlmodelc resources in the AIEngine package. Metal shaders compiled as .metal files in CaptureEngine. No external dependency manager — all third-party code vendored or avoided in favor of Apple frameworks.

## 🧠 Backend Architecture
- **Approach:** No server backend. All processing is local. The app is organized as a set of Swift actor-based services coordinated through Structured Concurrency. Each service exposes an async protocol interface. A central AppServices actor manages service lifecycle, initialization order, and dependency injection. Background work (OCR indexing, AI inference, video encoding) runs in detached tasks with cooperative cancellation via TaskGroup and withTaskCancellationHandler.
- **API Style:** Internal protocol-based async APIs between services. External automation via three surfaces: (1) AppIntents framework for Shortcuts actions — each capture type and post-action is a separate AppIntent struct. (2) x-callback-url scheme (snapforge://) parsed by a URLRouter that maps paths to service calls. (3) Local HTTP bridge API on localhost:48721 using NWListener with JSON request/response — RESTful routes mapping to service methods. The HTTP API requires a local bearer token stored in Keychain, generated on first launch. Plugin system uses a Swift protocol (SendToPlugin) with async methods for authentication, upload, and link generation.
- **Services:**
- CaptureService — Wraps ScreenCaptureKit for all capture types (screenshot, scrolling, video, GIF, timed). Manages SCStream lifecycle, region selection, pixel snapping, and spacing hints. Outputs raw capture data as SendableImage or CMSampleBuffer streams. Actor-isolated to serialize capture state transitions.
- AIInferenceService — Pluggable inference behind a unified InferenceProvider protocol with methods: func generate(prompt:context:) -> AsyncThrowingStream<String, Error>. Concrete providers: CoreMLProvider (Vision framework + Core ML models for OCR, region detection, Explain Screenshot), MLXProvider (MLX Swift for local LLM inference), OllamaProvider (HTTP client to local Ollama server), CloudAPIProvider (OpenAI/Anthropic REST APIs). Provider selection managed by user preference. All providers return AsyncThrowingStream for uniform streaming UI.
- ModelManagerService — Downloads, caches, validates, and loads AI models. Stores models in ~/Library/Application Support/SnapForge/Models/. Tracks model metadata (size, version, capabilities) in SwiftData. Manages GPU memory by loading/unloading models on demand. Exposes download progress as AsyncSequence. Handles MLX model conversion and quantization metadata.
- LibraryService — Manages the capture library backed by SwiftData. Stores CaptureRecord entities with metadata (timestamp, source app, bundle ID, domain, capture type, tags, file path, thumbnail path). Delegates full-text search to a parallel SQLite FTS5 index for OCR text. Provides filtered queries, smart filters, storage usage calculation, and auto-cleanup rule execution. Runs OCR indexing as background tasks after each capture.
- AnnotationService — Manages annotation state for the current capture. Provides tools (arrow, rectangle, oval, text, blur, highlight, numbering, crop). Renders annotations as a separate CALayer composited over the capture image. Supports undo/redo via a command pattern stack. AI annotation presets request highlight regions from AIInferenceService and apply them automatically.
- SharingService — Handles post-capture sharing with explicit privacy mode (LocalOnly, Upload, AskEveryTime). For uploads: E2E encrypts capture data client-side using CryptoKit (AES-256-GCM) before transmitting to configured storage backend. Supports self-hosted S3-compatible endpoints. Generates shareable links with optional expiry and password protection. Manages upload queue with retry logic.
- AutomationBridgeService — Exposes app functionality to external automation. Registers Shortcuts actions (NSUserActivity + AppIntents) for every capture type and post-action. Handles x-callback-url schemes via NSAppleEventManager. Runs a local HTTP server (NWListener on localhost) as the HTTP bridge API for scripting. Manages the plugin system: plugins are Swift packages conforming to a SendToPlugin protocol, loaded at runtime from a plugins directory.
- PromptStorageService — Stores and manages prompt templates and conversation history for AI features. Uses SwiftData with PromptTemplate and ConversationEntry models. Tracks per-prompt usage statistics (invocation count, average latency, token usage). Provides template CRUD and conversation replay for Explain Screenshot history.
- PreferencesService — Strongly-typed wrapper around UserDefaults and @AppStorage. Manages all user preferences: capture shortcuts, post-capture action memory per mode, sharing mode, AI provider selection, recording quality presets, library cleanup rules, and progressive disclosure state (seen tips, completed tour).
- RecordingPipelineService — Metal-accelerated video/GIF recording pipeline using ScreenCaptureKit + AVFoundation + Metal. Configurable FPS, resolution, codec (H.264/H.265/ProRes), and bitrate per recording mode. Real-time frame processing via Metal compute shaders for effects. Monitors CPU/GPU usage via IOKit and adjusts quality automatically to prevent stuttering. Outputs to AVAssetWriter with trimming support.

## 🗄 Data Layer
- **Primary Store:** SwiftData with a single ModelContainer configured for local storage at ~/Library/Application Support/SnapForge/. Models: CaptureRecord (id, timestamp, captureType, sourceAppBundleID, sourceAppName, sourceDomain, filePath, thumbnailPath, ocrText, tags, annotations, sharingStatus, fileSize), PromptTemplate (id, name, template, category, usageCount, averageLatency), ConversationEntry (id, captureRecordID, prompt, response, provider, tokenCount, latency, timestamp), AIModel (id, name, providerType, filePath, fileSize, version, capabilities, isLoaded), PluginConfig (id, name, bundlePath, isEnabled, authToken). Optional CloudKit sync for CaptureRecord metadata only (not files) when user opts in. A parallel SQLite database with FTS5 virtual table stores OCR text indexed by CaptureRecord.id for full-text search — managed directly via sqlite3 C API wrapped in a Swift actor, not through SwiftData.
- **Relationships:** CaptureRecord is the central entity. ConversationEntry has a many-to-one relationship to CaptureRecord via captureRecordID. Tags are stored as a comma-separated string on CaptureRecord for simplicity (queried via LIKE and FTS5). Annotations are serialized as JSON data on CaptureRecord. AIModel is independent — referenced by provider type string in ConversationEntry. PluginConfig is independent. The FTS5 index is a separate SQLite table joined to CaptureRecord by id at query time.
- **Migrations:** SwiftData lightweight migration via VersionedSchema. Each schema version is a numbered Swift type (SnapForgeSchemaV1, V2, etc.) with a SchemaMigrationPlan defining sequential migration stages. The FTS5 SQLite database is versioned separately with a simple integer version table and migration SQL scripts applied at app launch. Destructive migrations are never used — all migrations are additive (new columns, new tables) with default values for backward compatibility.

## ☁️ Infrastructure
- **Hosting:** No hosting required for core functionality. The app is distributed via the Mac App Store and/or direct download with Sparkle for updates. For optional cloud sharing: users configure their own S3-compatible endpoint (AWS S3, Backblaze B2, MinIO self-hosted) in preferences. SnapForge provides a downloadable Docker image for self-hosted storage (a minimal Go/Swift server that stores encrypted blobs and serves redirect URLs). No SnapForge-operated cloud infrastructure at launch.
- **Scaling Strategy:** Local performance scaling only. SQLite FTS5 handles tens of thousands of captures with sub-200ms queries via proper indexing and PRAGMA optimizations (WAL mode, mmap). AI model loading uses lazy initialization — models loaded on first inference request and unloaded after configurable idle timeout to free GPU memory. Recording pipeline uses Metal compute shaders for parallel frame processing scaled to available GPU cores. Background OCR indexing is throttled via TaskGroup with limited concurrency (max 2 concurrent OCR tasks) to avoid starving foreground capture responsiveness. Large library pagination via SwiftData FetchDescriptor with fetchLimit and fetchOffset.
- **CI/CD:** Xcode Cloud or GitHub Actions with macOS runners. Build pipeline: (1) swift build for all SPM packages, (2) xcodebuild for the main app target, (3) swift test for unit and integration tests, (4) xcrun notarytool for notarization, (5) Sparkle appcast generation for direct distribution. Core ML models are pre-compiled (.mlmodelc) and stored in Git LFS, bundled at build time. Metal shaders compiled automatically by Xcode build system.

## ⚖️ Key Trade-offs
- SwiftData over raw SQLite for primary persistence: SwiftData provides automatic CloudKit sync, SwiftUI integration, and migration tooling at the cost of less query flexibility. The FTS5 requirement forces a parallel raw SQLite database for full-text search, adding complexity but keeping SwiftData's benefits for structured data.
- Pluggable AI provider protocol over hardcoded Core ML: Adding MLX, Ollama, and cloud API providers increases interface complexity but future-proofs the AI subsystem. The unified AsyncThrowingStream return type keeps the UI layer simple regardless of backend. The tradeoff is more provider implementations to maintain.
- Single-process monolith over XPC services: Running capture, AI inference, and recording in-process simplifies communication and state sharing but means a crash in any subsystem takes down the app. Accepted because Swift Structured Concurrency provides sufficient isolation and the app targets Apple Silicon where memory and CPU are abundant.
- Local SQLite FTS5 over a full search engine (Tantivy, Lucene): FTS5 is simpler, ships with macOS, and handles tens of thousands of documents well. It lacks ranking sophistication and fuzzy matching of dedicated engines, but keeps the dependency footprint at zero and avoids bundling native libraries.
- Comma-separated tags on CaptureRecord over a normalized Tag entity with join table: Simpler schema and faster writes for the common case of 1-5 tags per capture. Searching by tag uses FTS5 or LIKE queries. The tradeoff is no referential integrity and slightly slower tag-based filtering at very large scale, accepted for simplicity.
- Actor-isolated services over global singletons: Each major service is a Swift actor, providing data race safety by default. The tradeoff is potential actor hop overhead for frequent cross-service calls, mitigated by batching operations and using nonisolated methods where safe.
- Bundled Core ML models over runtime download: Shipping pre-compiled .mlmodelc in the app bundle ensures instant availability but increases app size. The ModelManagerService supports downloading additional models post-install for larger LLMs (MLX), keeping the base app under 100MB while enabling advanced AI features on demand.

## 📐 Non-Functional Requirements
- Capture latency under 100ms from shortcut press to region selection overlay appearing, measured from CGEvent timestamp to first SwiftUI frame render
- Library full-text search returns results in under 200ms across 10,000+ OCR-indexed captures using SQLite FTS5 with WAL mode
- Metal-accelerated recording maintains target FPS without exceeding 15% CPU overhead on Apple Silicon Macs, monitored via IOKit counters
- All AI inference runs on-device by default via Core ML and Apple Vision framework — no data leaves the device unless user explicitly selects a cloud API provider
- E2E encryption (AES-256-GCM via CryptoKit) for all cloud uploads with keys derived from user passphrase, never transmitted
- Zero captures uploaded to cloud without explicit user consent enforced by SharingService privacy mode toggle
- App must run fully offline with all core features functional — cloud features degrade gracefully with clear error states
- Memory footprint under 200MB for typical usage (library browsing, annotation). AI model loading may temporarily increase to 500MB-2GB depending on model size, with automatic unloading after idle timeout
- Swift 6 strict concurrency compliance — all services are Sendable, all mutable state is actor-isolated, no data races
- Minimum macOS 15 (Sequoia) on Apple Silicon or Intel Macs with Metal support
- Accessibility: full VoiceOver support for all UI elements, complete keyboard navigation for all workflows, Dynamic Type support in library and settings views
